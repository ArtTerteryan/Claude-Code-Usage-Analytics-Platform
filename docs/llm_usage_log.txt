I need you to build an end-to-end analytics platform for Claude Code telemetry data as part of an internship assignment for Provectus. Before writing any code, I need you to deeply understand the requirements and the dataset.

## Assignment Requirements
Build an analytics platform that:
1. Ingests and processes telemetry data from Claude Code sessions (JSONL format) + employee CSV
2. Stores data in a structured database (SQL preferred — SQLite for portability)
3. Extracts analytics: token consumption trends by role, peak usage times, tool usage patterns, cost analysis, error patterns
4. Presents insights via an interactive Streamlit dashboard
5. Includes error handling, data validation, and clean architecture

Bonus features (implement at least 2):
- Predictive analytics (trend forecasting or anomaly detection)
- Real-time data streaming simulation

- REST API endpoints for programmatic access

## Dataset Structure
Two files generated by `generate_fake_data.py`:
- `output/telemetry_logs.jsonl` — JSONL, each line is a CloudWatch-style log batch with `logEvents` array. Each event's `message` is a JSON string containing `body`, `attributes`, `scope`, `resource`
- `output/employees.csv` — columns: email, full_name, practice, level, location

Event types in the telemetry:
- `claude_code.api_request` — model, cost_usd, duration_ms, input_tokens, output_tokens, cache_read_tokens, cache_creation_tokens
- `claude_code.tool_decision` — tool_name, decision (accept/reject), source
- `claude_code.tool_result` — tool_name, success, duration_ms
- `claude_code.user_prompt` — prompt_length (content redacted)
- `claude_code.api_error` — error message, status_code, model, attempt

Common attributes across all events: user.email, session.id, event.timestamp, terminal.type, organization.id

Resource fields: host.arch, os.type, os.version, service.version

2. README.md with setup instructions, architecture overview, dependencies
3. Working Streamlit dashboard
4. LLM usage log section in README

PROMPT 2
-------------------------------------------------------------------------------
Based on your requirements analysis in `docs/01_requirements_analysis.md`, now design the system architecture. I want you to produce a detailed architecture document that I will review and approve before any coding begins.

## Design Constraints
- Python 3.11+, no Docker required for local dev
- SQLite for storage (portable, zero-config, good enough for this dataset size)
- Streamlit for the dashboard
- FastAPI for the optional API layer
- Use clean separation: data layer, business logic, presentation
- Follow the Repository pattern for database access
- Must handle the full dataset (~450K events, ~500MB JSONL) efficiently

## What I Need You To Design

### 1. Project Structure
Propose a complete file/folder tree. Think in layers:
- `src/ingestion/` — parsing, validation, loading
- `src/database/` — schema, models, repositories
- `src/analytics/` — query builders, statistical computations
- `src/api/` — FastAPI endpoints (bonus)
- `src/dashboard/` — Streamlit pages
- `tests/` — unit tests for critical paths
- `docs/` — documentation

### 2. Database Schema
Design normalized SQLite tables. Consider:
- A `sessions` table (derived from grouping events by session.id)
- An `api_requests` table (one row per api_request event)
- A `tool_events` table (combining decisions + results by matching tool_name within a session)
- An `errors` table
- An `employees` table
- Proper indexes for common query patterns (by date, by user, by model, by practice)
- Think about what should be pre-aggregated vs. queried on the fly

### 3. Ingestion Pipeline Design
- How to parse nested JSONL → flat records
- Batch insert strategy (not one-at-a-time)
- Data validation rules (what to reject, what to coerce)
- Progress reporting for large files

### 4. Dashboard Layout
- Which pages/tabs and what charts on each
- Which analytics questions each page answers
- Filtering strategy (date range, practice, level, model)

### 5. Bonus Features Plan
Pick 2-3 bonus features and describe how they will work:
- Anomaly detection: which algorithm, on which metrics
- Trend forecasting: what approach (e.g., simple linear regression, moving averages)
- API design: which endpoints, what response format

Save the complete architecture document to `docs/02_architecture.md`. Include the full file tree, all table schemas with column types, and the dashboard wireframe descriptions.

## 4A: Database Schema & Models
Implement the database module:
- Create all SQLite tables using the schema from the architecture doc
- Use raw SQL with Python's `sqlite3` module (no ORM — keeps it simple and fast)
- Create a `DatabaseManager` class that handles connection lifecycle and provides methods for:
  - `initialize()` — create tables and indexes
  - `execute()` / `executemany()` — with proper error handling
  - Context manager support (`with` statement)
- Create Repository classes for each table with insert_batch() and common query methods
- Add proper indexes as specified in the architecture

## 4B: Ingestion Pipeline
Implement the ingestion module:
- Parse JSONL file line by line (memory efficient — don't load the whole file)
- For each log batch, iterate through `logEvents`, parse the JSON `message`
- Validate each event: check required fields exist, coerce string numbers to proper types
- Route events to the correct table based on `body` field
- Use batch inserts (collect 1000 records, then INSERT)
- Add a progress bar (use `tqdm` or just print progress every N batches)
- Handle malformed records gracefully — log warnings, don't crash
- Also ingest `employees.csv`
